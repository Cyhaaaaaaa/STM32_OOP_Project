# 第一部分：架构深度分析

## 1.1 目录安排的核心思想

### 1.1.1 核心思想：**依赖方向单向流动，分层即分责**

```
┌──────────────────────────────────────────────────────┐
│  App/          ← 业务逻辑层                          │
│  "绿灯 300ms 闪一次，红灯 100ms 闪一次"               │
│  零硬件依赖，纯逻辑代码                               │
├──────────────────────────────────────────────────────┤
│  Devices/      ← 设备抽象层                          │
│  "这个板子上有一盏绿灯、一盏红灯"                     │
│  只依赖 BSP 通用接口，不碰芯片寄存器                   │
├──────────────────────────────────────────────────────┤
│  BSP/          ← 板级支持包（硬件隔离墙）              │
│  "GPIO 引脚怎么操作"                                  │
│  全村唯一 #include <芯片头文件> 的地方                 │
├──────────────────────────────────────────────────────┤
│  System/       ← 系统服务层                          │
│  "时间基准、中断服务"                                 │
│  封装 SysTick 等内核级服务                            │
├──────────────────────────────────────────────────────┤
│  Driver/       ← 芯片厂商外设库                       │
│  CMSIS/        ← ARM Cortex 内核接口                 │
│  直接替换，零修改                                     │
└──────────────────────────────────────────────────────┘
```

**核心思想可以用四个词概括：**

| 思想 | 本工程的体现 |
|------|-------------|
| **依赖倒置** | App 不依赖芯片，两者都依赖 BSP 抽象接口 |
| **接口隔离** | 每层只暴露最小接口，`device_led.h` 暴露 7 个函数，`bsp_gpio.h` 暴露 4 个操作 |
| **单一职责** | `delay.c` 只管时间、`bsp_gpio.c` 只管 GPIO、`device_led.c` 只管 LED 映射 |
| **开闭原则** | 加新设备只需新增文件；换芯片上层零修改 |

### 1.1.2 目录与依赖的物理对应

目录结构直接反映了依赖规则——**同一个目录内的文件互相可见，跨目录只能通过 `.h` 接口通信**：

```
app_lighting.c ──→ device_led.h ──→ bsp_gpio.h        ← 纯净接口链
       │                                    │
       └────────── delay.h ────────────────┘
                       │
                       ├── stm32f10x.h (芯片寄存器定义)
                       │
    bsp_gpio.c ────────┘  ← 全村唯一的芯片头文件引用
    bsp_uart.c ────────┘
    delay.c ───────────┘  ← 需要 SysTick 寄存器
    stm32f10x_it.c ─────┘  ← 中断向量，不可避免
```

**关键约束**：`device_led.c` 和 `app_lighting.c` 的 include 链中**绝不出现** `stm32f10x.h`。这不是巧合，是刻意设计的结果。这个约束保证了上层代码的硬件无关性。

### 1.1.3 为什么不是简单的 `src/` + `inc/` 扁平结构？

传统的嵌入式项目目录通常长这样：

```
Project/
├── src/
│   ├── main.c
│   ├── led.c        ← GPIO 寄存器、延时、LED 逻辑全部混在一起
│   ├── uart.c
│   └── delay.c
└── inc/
    ├── led.h
    ├── uart.h
    └── delay.h
```

这种扁平结构在 3 个文件时没问题。但当项目发展到 50+ 个源文件时，问题会集中爆发：

- **不知道谁依赖谁**：`sensor.c` 能不能直接调用 `GPIO_SetBits()`？没人说得清
- **换芯片 = 全项目大扫除**：`GPIOC` 散落在 30 个文件里，逐个修改，逐个测试
- **代码复用靠复制粘贴**：从项目 A 复制 `led.c` 到项目 B，发现里面写着 `RCC_APB2Periph_GPIOC`，在项目 B 的芯片上根本不存在

**本工程的方案**：用目录结构强制分层，用物理隔离来保证逻辑隔离。你想在 `Devices/` 层直接操作寄存器？编译器不让你过，因为 `.h` 里只有不透明指针 `struct GPIO_Pin;`，根本没有暴露寄存器地址。

## 1.2 软件结构的设计意图

### 1.2.1 设计意图总览

| 层次 | 设计意图 | 类比 |
|------|---------|------|
| **App/** | 表达"产品要做什么"，不关心"怎么做到" | 产品经理的需求文档 |
| **Devices/** | 定义"板子上有什么"，用配置单描述硬件 | 硬件工程师的 BOM 表 |
| **BSP/** | 实现"芯片怎么操作"，封装寄存器细节 | 驱动工程师的数据手册翻译 |
| **System/** | 提供"时间怎么管理"，统一时间基准 | 操作系统的时钟服务 |
| **Driver/CMSIS** | 芯片厂商提供的"原子能力" | 芯片出厂说明书 |

### 1.2.2 设计意图的关键实现手段

#### 手段一：不透明指针（Opaque Pointer）—— C 语言的 `private`

```c
// bsp_gpio.h —— 头文件只声明，不定义
struct GPIO_Pin;  // 前向声明：全项目除了 bsp_gpio.c 没人知道里面有什么

// bsp_gpio.c —— 定义藏在 .c 文件里
struct GPIO_Pin {
    GPIO_TypeDef* port;   // 这两个字段对外界是"不可见"的
    uint16_t      pin;
};
```

这不是语法层面的 `private`（C 语言没有这个概念），而是**物理层面的封装**——其他 `.c` 文件就算想看也看不到。任何绕过 `GPIO_Toggle()` 接口直接操作寄存器的企图，都会在编译阶段被阻止。

#### 手段二：静态内存池 —— 嵌入式友好的"对象管理"

```c
struct GPIO_Pin* GPIO_Pin_Create(GPIO_Config_t* cfg) {
    static struct GPIO_Pin pool[16];   // 编译期确定大小
    static uint8_t pool_idx = 0;

    if (pool_idx >= 16) return NULL;   // 满了就优雅拒绝
    struct GPIO_Pin* obj = &pool[pool_idx++];
    // ... 初始化 ...
    return obj;
}
```

**设计意图**：在资源受限的 MCU（RAM 通常只有 20KB~64KB）上，`malloc` 是奢侈品也是定时炸弹——内存碎片可能在运行数周后导致申请失败。静态池子让内存占用在编译时就能确定，`Linker` 的 `.map` 文件直接告诉你用了多少 RAM。

#### 手段三：翻译层模式 —— 把"芯片方言"关在笼子里

```c
// 内部私有函数：把通用枚举翻译成具体的 STM32 寄存器地址
static void BSP_GPIO_Translate(BSP_Port_t bsp_port, BSP_Pin_t bsp_pin,
                               GPIO_TypeDef** out_port, uint16_t* out_pin,
                               uint32_t* out_rcc) {
    switch (bsp_port) {
        case BSP_PORT_A: *out_port = GPIOA; *out_rcc = RCC_APB2Periph_GPIOA; break;
        case BSP_PORT_C: *out_port = GPIOC; *out_rcc = RCC_APB2Periph_GPIOC; break;
        // ...
    }
    *out_pin = (uint16_t)(1 << bsp_pin);
}
```

**设计意图**：芯片手册上的寄存器名字（`GPIOC`、`RCC_APB2Periph_GPIOC`、`GPIO_Pin_13`）被打包在一个 `static` 函数里，对外暴露的是 `BSP_PORT_C`、`BSP_PIN_13` 这样的通用名字。换芯片时，只需改这个翻译函数，所有上层代码纹丝不动。

#### 手段四：非阻塞时间管理 —— 无 RTOS 的多任务并发

```c
// 超级循环 + 非阻塞超时检查 = 协作式多任务
while (1) {
    if (systick_timeout(&t1, 300)) { /* 任务A：300ms 周期 */ }
    if (systick_timeout(&t2, 100)) { /* 任务B：100ms 周期 */ }
    if (systick_timeout(&t3, 50))  { /* 任务C：50ms 周期 */  }
    // 所有任务看起来是"并行"的，实际上是在 1ms 级的时间片上轮流执行
}
```

**设计意图**：在不需要 RTOS 开销（RAM、Flash、上下文切换延迟）的前提下，实现多任务并发。这对资源紧张的 MCU 尤其重要——你不必为了跑两个 LED 和一个传感器而引入一个 4KB RAM 开销的 FreeRTOS。

#### 手段五：`main.c` 作为组装清单，不做具体工作

```c
int main(void) {
    NVIC_PriorityGroupConfig(NVIC_PriorityGroup_2);  // 全局配置
    Delay_Init();                                     // 系统服务
    App_Lighting_Init();                              // 业务组件1
    // App_WeightSensor_Init();                       // 业务组件2（未来）
    while (1) {
        App_Lighting_Task();
        // App_WeightSensor_Task();
    }
}
```

**设计意图**：`main.c` 不需要理解任何业务细节，它只是一个**组装清单**——类似依赖注入框架里的 `Composition Root`。这种写法让 `main.c` 天然成为系统的目录页和初始化顺序文档。

## 1.3 维护性分析

### 1.3.1 维护性评分：★★★★★（优秀）

### 1.3.2 场景验证

#### 场景 A：修改 LED 闪烁频率

| 对比维度 | 传统写法 | 本工程 |
|---------|---------|--------|
| 需要改的文件 | 1 个（但里面有 500 行业务+硬件混杂代码） | 1 个（`app_lighting.c`，改一个数字） |
| 需要理解的内容 | 需要理解 GPIO 寄存器配置 + 延时逻辑 + 业务逻辑 | 只需要理解 `300` 这个数字的含义 |
| 引入 bug 的风险 | 高（可能在改频率时不小心动了 GPIO 初始化） | 极低（只改一个常量） |

```c
// 维护操作：把绿灯从 300ms 改为 500ms
// 修改前：
if (systick_timeout(&led_green_timer, 300)) { ... }
// 修改后：
if (systick_timeout(&led_green_timer, 500)) { ... }
// 一行改动，影响范围完全可控
```

#### 场景 B：更换 LED 连接的 GPIO 引脚

| 对比维度 | 传统写法 | 本工程 |
|---------|---------|--------|
| 需要改的文件 | 3~5 个（初始化代码、中断配置、业务逻辑中散落的引脚引用） | 1 个（`device_led.c`，改一个配置结构体） |
| 需要全局搜索 | 必须，搜索 `GPIOC` 和 `GPIO_Pin_13` | 不需要，看一眼配置单就行 |

```c
// 维护操作：绿灯从 PC13 改到 PB0
// 修改前：
static GPIO_Config_t led_green_cfg = {
    .port = BSP_PORT_C, .pin = BSP_PIN_13, .mode = BSP_GPIO_MODE_OUT_PP
};
// 修改后：
static GPIO_Config_t led_green_cfg = {
    .port = BSP_PORT_B, .pin = BSP_PIN_0, .mode = BSP_GPIO_MODE_OUT_PP
};
// 改两个枚举值，其他所有文件不感知
```

#### 场景 C：增加一个新设备（如按键）

需要新增的文件：

| 新文件 | 做什么 |
|--------|--------|
| `Devices/device_key.h` | 按键设备接口声明 |
| `Devices/device_key.c` | 按键设备实现（配置 + 消抖 + 状态读取） |
| `App/app_key.h` | 按键业务逻辑接口 |
| `App/app_key.c` | 按键业务逻辑（短按/长按/组合键） |

需要修改的文件：**仅 `App/main.c`** —— 加两行（`Init` + `Task`）。

已有的 `bsp_gpio.h/c` 不需要任何修改，因为 `GPIO_Pin_Create()` 已经支持输入模式的配置。

#### 场景 D：增加一个新的 BSP 外设（如 SPI）

需要新增的文件：

| 新文件 | 做什么 |
|--------|--------|
| `BSP/bsp_spi.h` | SPI 抽象接口（通用枚举 + 配置结构体 + 不透明指针） |
| `BSP/bsp_spi.c` | SPI 实现（翻译层 + 初始化 + 读写操作） |

对已有代码的影响：**零**。App 层和 Devices 层的代码一行不动。

### 1.3.3 维护性结论

| 维护操作类型 | 影响范围 | 风险等级 |
|-------------|---------|---------|
| 修改业务逻辑（如闪烁频率） | 单文件，单函数 | 极低 |
| 修改硬件接线（如更换引脚） | 单文件，单结构体 | 极低 |
| 增加新设备 | 新增 2~4 个文件 + main.c 加两行 | 低 |
| 增加新 BSP 外设 | 新增 2 个文件 | 低 |
| 更换芯片平台 | BSP 层 + System 层 + Driver 层 | 中（但隔离得很好） |
| 修改系统时钟频率 | `system_stm32f10x.c` 一个 define | 极低 |

**核心优势**：每一类变更都有明确的"入口文件"，不需要在整个项目中大海捞针。这是分层架构对维护性最直接的贡献。

## 1.4 通用性分析

### 1.4.1 通用性评分：★★★★☆（良好）

### 1.4.2 可复用的层次

| 层次 | 通用性 | 可复用到什么程度 |
|------|--------|-----------------|
| **App/** | ★★★★★ | **完全通用**。不包含任何硬件依赖，理论上可以不做任何修改在 Linux、Windows、任何 MCU 上编译通过 |
| **Devices/** | ★★★★★ | **完全通用**。只依赖 `bsp_gpio.h`（纯接口头文件），不依赖任何芯片头文件 |
| **BSP/bsp_*.h** | ★★★★☆ | **接口通用**。`BSP_Port_t`、`BSP_Pin_t`、`GPIO_Config_t` 这些定义与芯片无关。唯一需要扩展的是端口枚举（不同芯片的 GPIO 端口数量不同） |
| **BSP/bsp_*.c** | ★☆☆☆☆ | **芯片专属**。这是设计目标——所有的芯片相关性都集中在这里 |
| **System/delay.h** | ★★★★☆ | **接口通用**。`systick_timeout()` 接口在任何 MCU 上都有意义 |
| **System/delay.c** | ★★☆☆☆ | **部分通用**。SysTick 是 Cortex-M 的特性，非 ARM 芯片需要替换实现 |
| **Driver/FWLIB/** | ★☆☆☆☆ | **芯片专属**。原样来自芯片厂商 |
| **CMSIS/** | ★★☆☆☆ | **架构专属**。Cortex-M3 通用（同核心的芯片可复用） |

### 1.4.3 通用性的关键设计要素

#### 要素一：`bsp_gpio.h` 中枚举值的平台无关性

```c
typedef enum {
    BSP_PORT_A, BSP_PORT_B, BSP_PORT_C, BSP_PORT_D,
    // 换大容量芯片，在这里扩展 BSP_PORT_E, BSP_PORT_F...
} BSP_Port_t;

typedef enum {
    BSP_PIN_0 = 0,  BSP_PIN_1,  BSP_PIN_2,  BSP_PIN_3,
    BSP_PIN_4,      BSP_PIN_5,  BSP_PIN_6,  BSP_PIN_7,
    BSP_PIN_8,      BSP_PIN_9,  BSP_PIN_10, BSP_PIN_11,
    BSP_PIN_12,     BSP_PIN_13, BSP_PIN_14, BSP_PIN_15
} BSP_Pin_t;
```

这些枚举值不做任何平台假设——它们只是表达"第 0 号引脚"、"第 1 号引脚"这样的通用概念。至于在具体的芯片上"第 0 号引脚"实际对应哪个寄存器位，那是 BSP `.c` 文件的翻译函数该关心的事。

#### 要素二：`UART_Config_t` 的通用配置语言

```c
typedef struct {
    BSP_UART_t          uart;       // UART1/2/3 的抽象编号
    uint32_t            baudrate;   // 标准波特率，所有芯片通用
    BSP_UART_DataBits_t data_bits;  // 8/9 位，所有芯片通用
    BSP_UART_StopBits_t stop_bits;  // 1/0.5/2/1.5 位，所有芯片通用
    BSP_UART_Parity_t   parity;     // 无/奇/偶校验，所有芯片通用
} UART_Config_t;
```

这个结构体里的每一项都是用"工程师的自然语言"表达的，不是用芯片的宏名表达的。它在任何 MCU 上都有意义。

### 1.4.4 通用性的局限

当前框架的通用性有一个边界条件：**它假设底层有类似 GPIO 引脚 + 外设时钟的概念**。对于某些高度抽象的平台（如 Arduino API 封装过的 ESP32），BSP 层的 `GPIO_TypeDef*` 结构体定义需要重新设计，但 `bsp_gpio.h` 的接口保持不变。

**当前框架最适合的是**：所有使用 C 语言、有 GPIO 概念、以超级循环 + 中断为运行模型的嵌入式 MCU（涵盖市场上 90% 以上的 MCU）。

## 1.5 阅读性分析

### 1.5.1 阅读性评分：★★★★★（优秀）

### 1.5.2 逐层阅读路径

一个新人接手这个项目时，阅读路径非常清晰：

```
第 1 步：打开 main.c
  → 3 秒内理解系统有哪些组件（Init + Task 列表就是目录）

第 2 步：打开 app_lighting.c
  → 10 秒内理解业务逻辑："绿灯 300ms 闪，红灯 100ms 闪"
  → 不需要理解任何硬件细节

第 3 步：打开 device_led.c
  → 15 秒内理解设备映射："绿灯在 PC13，红灯在 PA1"
  → 不需要理解 GPIO 寄存器如何操作

第 4 步：打开 bsp_gpio.c
  → 只在需要调试寄存器级问题时才需要看
  → 包含了所有硬件细节，但它们是"关在笼子里"的
```

### 1.5.3 提升阅读性的具体手段

#### 手段一：C99 指定初始化器 —— 自文档化的配置

```c
// 传统写法（需要查结构体定义才知道每个值是什么）：
GPIO_Config_t cfg = { BSP_PORT_C, BSP_PIN_13, BSP_GPIO_MODE_OUT_PP };

// 本工程写法（不需要查任何东西，一眼就懂）：
static GPIO_Config_t led_green_cfg = {
    .port = BSP_PORT_C,
    .pin  = BSP_PIN_13,
    .mode = BSP_GPIO_MODE_OUT_PP
};
```

#### 手段二：语义化函数命名 —— 读代码像读需求

```c
// App 层的函数名表达"意图"
App_Lighting_Task();        // 灯光控制任务

// Devices 层的函数名表达"目标设备 + 动作"
Device_LED_Green_Toggle();  // 绿灯-翻转
Device_LED_Red_On();        // 红灯-亮

// BSP 层的函数名表达"通用操作"
GPIO_Toggle(pin);           // GPIO-翻转
GPIO_SetHigh(pin);          // GPIO-置高

// System 层的函数名表达"服务语义"
systick_timeout(&t, 300);   // 系统节拍-超时检查
State_Keep_Timeout(&st, v, 20);  // 状态保持-超时检测（消抖）
```

# 第二部分：移植指南

## 2.1 场景一：移植到 GD32F103

### 2.1.1 背景分析

**GD32F103** 是兆易创新（GigaDevice）推出的基于 **ARM Cortex-M3** 内核的 MCU，在引脚和封装上与 STM32F103 **完全兼容**（pin-to-pin 替代）。两者都使用 ARM Cortex-M3 内核，这意味着：

| 对比维度 | STM32F103 | GD32F103 | 兼容性 |
|---------|-----------|----------|--------|
| **CPU 内核** | Cortex-M3 | Cortex-M3 | ✅ 完全相同 |
| **启动文件** | `startup_stm32f10x_md.s` | `startup_gd32f10x_md.s` | ⚠️ 向量表兼容，但需替换 |
| **系统时钟** | 8MHz HSE, PLL 到 72MHz | 8MHz HSE, PLL 到 108MHz | ⚠️ 时钟初始化不同 |
| **GPIO 寄存器** | `GPIOA->ODR` 等 | `GPIOA->ODR` 等 | ✅ 寄存器名相同 |
| **时钟控制** | `RCC_APB2Periph_GPIOA` | `RCU_GPIOA`（RCC 改名 RCU） | ⚠️ 宏名不同 |
| **USART 寄存器** | `USART1->DR` | `USART1->DR` | ✅ 寄存器名相同 |
| **SysTick** | 24 位递减计数器 | 24 位递减计数器 | ✅ 完全相同 |
| **NVIC** | Cortex-M3 标准 | Cortex-M3 标准 | ✅ 完全相同 |
| **标准库函数** | `GPIO_Init()` | `gpio_init()` | ⚠️ 函数名风格不同 |
| **主频** | 72MHz | 108MHz（同频时 Flash 等待周期不同） | ⚠️ |

**核心结论**：GD32F103 是最接近 STM32F103 的替代方案，属于**同架构、同外设、同引脚**移植。移植工作主要集中在替换底层库文件 + 修改 BSP 翻译层的宏名映射。

#### 步骤 3：修改 `BSP/bsp_gpio.c` —— 核心工作

```c
// ==========================================
// bsp_gpio.c —— GD32F103 版本的修改点
// ==========================================

// --- 修改 1：替换头文件 ---
// 旧：#include "stm32f10x.h"
// 新：
#include "gd32f10x.h"

// --- 修改 2：结构体定义可能需要调整 ---
// GD32 的 GPIO 寄存器结构与 STM32 高度兼容，但类型名可能不同
// 如果 GD32 库的 GPIO 基地址类型不是 GPIO_TypeDef*，调整这里：
struct GPIO_Pin {
    uint32_t      port;    // GD32 有些版本用 uint32_t 表示基地址
    uint16_t      pin;
};
// 保守做法：检查 GD32 的 gpio.h，确认 GPIO 寄存器结构体的类型名

// --- 修改 3：翻译函数 —— 核心修改 ---
static void BSP_GPIO_Translate(BSP_Port_t bsp_port, BSP_Pin_t bsp_pin,
                               uint32_t* out_port, uint16_t* out_pin, uint32_t* out_rcc) {
    switch (bsp_port) {
        case BSP_PORT_A:
            *out_port = GPIOA;           // GD32 的宏名一般与 STM32 相同
            *out_rcc = RCU_GPIOA;        // ← 关键！GD32 将 RCC 改为 RCU
            break;
        case BSP_PORT_B:
            *out_port = GPIOB;
            *out_rcc = RCU_GPIOB;        // ← RCC_APB2Periph_GPIOB → RCU_GPIOB
            break;
        case BSP_PORT_C:
            *out_port = GPIOC;
            *out_rcc = RCU_GPIOC;        // ← 同理
            break;
        case BSP_PORT_D:
            *out_port = GPIOD;
            *out_rcc = RCU_GPIOD;
            break;
        default:
            break;
    }
    *out_pin = (uint16_t)(1 << bsp_pin); // 如果 GD32 的 GPIO_PIN_X 宏也是 (1<<n)，不变
}

// --- 修改 4：构造函数中初始化方式调整 ---
struct GPIO_Pin* GPIO_Pin_Create(GPIO_Config_t* cfg) {
    static struct GPIO_Pin pool[16];
    static uint8_t pool_idx = 0;

    if (pool_idx >= 16) return 0;

    struct GPIO_Pin* obj = &pool[pool_idx++];
    uint32_t rcc_clock = 0;

    BSP_GPIO_Translate(cfg->port, cfg->pin, &obj->port, &obj->pin, &rcc_clock);

    // --- 开启时钟：函数名变化 ---
    // 旧：RCC_APB2PeriphClockCmd(rcc_clock, ENABLE);
    // 新：
    rcu_periph_clock_enable(rcc_clock);   // GD32 的风格

    // --- 配置底层结构体 ---
    // GD32 的 GPIO 初始化结构体成员名可能不同
    gpio_init_para gi;                    // 旧：GPIO_InitTypeDef
    gi.gpio_pin   = obj->pin;            // 旧：gi.GPIO_Pin
    gi.gpio_speed = GPIO_SPEED_50MHZ;    // 旧：gi.GPIO_Speed
    gi.gpio_mode  = GPIO_MODE_OUT_PP;    // 旧：gi.GPIO_Mode

    switch(cfg->mode){
        case BSP_GPIO_MODE_OUT_PP:
            gi.gpio_mode = GPIO_MODE_OUT_PP;
            break;
        case BSP_GPIO_MODE_IN_PULLUP:
            gi.gpio_mode = GPIO_MODE_IPU;
            break;
        case BSP_GPIO_MODE_IN_FLOATING:
            gi.gpio_mode = GPIO_MODE_IN_FLOATING;
            break;
    }

    gpio_init(obj->port, &gi);           // 旧：GPIO_Init()

    // 默认高电平
    if(cfg->mode == BSP_GPIO_MODE_OUT_PP) {
        gpio_bit_set(obj->port, obj->pin); // 旧：GPIO_SetBits()
    }

    return obj;
}

// --- 修改 5：操作接口调整 ---
void GPIO_SetHigh(struct GPIO_Pin* pin) {
    if (pin) gpio_bit_set(pin->port, pin->pin);    // 旧：GPIO_SetBits()
}

void GPIO_SetLow(struct GPIO_Pin* pin) {
    if (pin) gpio_bit_reset(pin->port, pin->pin);  // 旧：GPIO_ResetBits()
}

void GPIO_Toggle(struct GPIO_Pin* pin) {
    if (pin) {
        // GD32 有专用的位翻转寄存器
        gpio_bit_toggle(pin->port, pin->pin);
        // 或直接操作 ODR 寄存器（如果 GD32 的 GPIO 结构体有 ODR 成员）：
        // (*(volatile uint32_t*)(pin->port + 0x0C)) ^= pin->pin;
    }
}

uint8_t GPIO_GetLevel(struct GPIO_Pin* pin) {
    if (!pin) return 0;
    // 旧：pin->port->IDR & pin->pin
    return gpio_input_bit_get(pin->port, pin->pin) ? 1 : 0;
}
```

#### 步骤 4：修改 `BSP/bsp_uart.c`

GD32 的 USART 库函数名变化对照：

| `USART_ReceiveData()` | `usart_data_receive()` |
| `USART_GetFlagStatus()` | `usart_flag_get()` |
| `USART_GetITStatus()` | `usart_interrupt_flag_get()` |

## 2.2 场景二：移植到 ESP32

### 2.2.1 背景分析

**ESP32** 是乐鑫科技（Espressif）推出的双核 Wi-Fi/蓝牙 MCU，使用 **Xtensa LX6** 或 **RISC-V** 处理器内核，与 STM32F103 的 Cortex-M3 有本质差异：

| 对比维度 | STM32F103 | ESP32 | 兼容性 |
|---------|-----------|-------|--------|
| **CPU 内核** | ARM Cortex-M3 | Xtensa LX6 双核 或 RISC-V 单核 | ❌ 完全不同 |
| **主频** | 72MHz | 240MHz | ❌ 完全不同 |
| **RAM** | 20KB SRAM | 520KB SRAM | ↑ 数量级差异 |
| **Flash** | 64KB~128KB 内置 | 4MB~16MB 外部 SPI Flash | ❌ 架构完全不同 |
| **开发框架** | STM32 标准外设库 / HAL | ESP-IDF（FreeRTOS + 丰富组件） | ❌ 完全不同 |
| **GPIO 控制** | 寄存器直接操作 | `gpio_set_level()` API | ⚠️ 思想相同，方式不同 |
| **SysTick** | 内置 24 位递减计数器 | FreeRTOS `xTaskGetTickCount()` | ⚠️ 需适配 |
| **UART** | USART 外设寄存器 | `uart_write_bytes()` API | ⚠️ 思想相同，方式不同 |
| **编译工具链** | arm-none-eabi-gcc | xtensa-esp32-elf-gcc | ❌ 完全不同 |
| **构建系统** | Keil MDK / Makefile | CMake + idf.py | ❌ 完全不同 |
| **中断管理** | NVIC（Cortex-M 标准） | 中断分配器 + FreeRTOS 任务通知 | ❌ 完全不同 |
| **多任务** | 裸机超级循环 | FreeRTOS（ESP-IDF 自带） | ❌ 完全不同 |

**核心结论**：这是**跨架构、跨平台、跨开发框架**的移植。但本工程的分层架构在此时发挥最大价值——App 层和 Devices 层**完全不需要修改**，只需要替换 BSP 层和 System 层的实现，并引入 ESP-IDF 框架替代原有的裸机运行时。

### 2.2.2 移植策略：保持上层接口，重写底层实现

移植的核心思想发生了根本变化：

```
STM32 版本的核心假设：                ESP32 版本的核心假设：
  裸机运行                            FreeRTOS 运行
  SysTick 1ms 中断驱动时间              FreeRTOS Tick 驱动时间
  寄存器直接操作 GPIO                   ESP-IDF API 操作 GPIO
  超级循环调度                          FreeRTOS 任务调度
  静态内存池                           FreeRTOS 动态内存（或保持静态池）
  Keil MDK 构建                        CMake + idf.py 构建
```

**但关键是**：`device_led.c` 和 `app_lighting.c` 看不到任何这些差异！它们只看到 `GPIO_Pin*` 不透明指针和 `systick_timeout()` 函数调用。

```c
// delay.c —— ESP32 版本
// 关键变化：用 FreeRTOS Tick 替代 SysTick 硬件计数器

#include "delay.h"
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

// --- 全局变量 ---
// ESP32 上不需要 volatile —— FreeRTOS 的 xTaskGetTickCount() 自带内存屏障
static uint32_t sys_tick_ms = 0;
static uint32_t sys_tick_us_fac = 0;

// --- Delay_Init ---
void Delay_Init(void) {
    // ESP-IDF 在 app_main() 之前已经初始化了 FreeRTOS
    // SysTick 不再需要配置，FreeRTOS 的 Tick 由硬件定时器组驱动
    // 但我们保留这个函数以兼容上层调用
    sys_tick_ms = 0;
    sys_tick_us_fac = 240; // ESP32 主频 240MHz，1us = 240 cycles（用于 us 延时）
}

// --- Delay_IncTick ---
// ESP32 版本不再需要中断驱动！
// 改为在 app_main() 的主循环或 FreeRTOS 定时器中调用，或直接用 xTaskGetTickCount()

// --- Get_SysTick ---
uint32_t Get_SysTick(void) {
    // 关键变化：用 FreeRTOS Tick 替代 SysTick
    // pdMS_TO_TICKS(1) 通常返回 10（100Hz Tick），一个 Tick = 10ms
    // 所以需要转换为毫秒
    return xTaskGetTickCount() * portTICK_PERIOD_MS;
}

// --- Delay_us ---
void Delay_us(uint32_t nus) {
    // ESP-IDF 提供精确的微秒延时 API
    ets_delay_us(nus);  // ROM 函数，比 FreeRTOS 延时更精确
}

// --- Delay_ms ---
void Delay_ms(uint32_t nms) {
    // 使用 FreeRTOS 的阻塞延时（让出 CPU，不忙等！）
    vTaskDelay(pdMS_TO_TICKS(nms));
}

// --- systick_timeout ---
// 核心算法完全不变！但依赖 Get_SysTick() 的时间基准
bool systick_timeout(uint32_t *timestamp, uint32_t timeout_ms) {
    uint32_t now = Get_SysTick();
    if (now - *timestamp >= timeout_ms) {
        *timestamp = now;
        return true;
    }
    return false;
}

// --- State_Keep_Timeout ---
// 完全不变！
bool State_Keep_Timeout(State_Timer_t *timer, uint8_t current_state, uint32_t timeout_ms) {
    if (timer->is_first == false || timer->last_state != current_state) {
        timer->last_state = current_state;
        timer->timestamp = Get_SysTick();
        timer->is_first = true;
        return false;
    }
    if (Get_SysTick() - timer->timestamp >= timeout_ms) {
        return true;
    }
    return false;
}
```

**关键变化分析**：

| 函数 | STM32 实现 | ESP32 实现 | 接口兼容性 |
|------|-----------|-----------|-----------|
| `Delay_Init()` | 配置 SysTick 产生 1ms 中断 | 空操作（FreeRTOS 已初始化） | ✅ 相同调用 |
| `Get_SysTick()` | 返回 `sys_tick_ms`（中断驱动） | 返回 `xTaskGetTickCount() * portTICK_PERIOD_MS` | ✅ 相同返回值语义 |
| `systick_timeout()` | 基于 `sys_tick_ms` | 基于 `Get_SysTick()` | ✅ 完全不变 |
| `State_Keep_Timeout()` | 基于 `sys_tick_ms` | 基于 `Get_SysTick()` | ✅ 完全不变 |
| `Delay_ms()` | 忙等循环 | `vTaskDelay()`（让出 CPU！） | ✅ 相同调用，实际更好 |
| `Delay_us()` | 读 SysTick 计数器 | `ets_delay_us()` | ✅ 相同调用 |

**注意**：ESP32 上 `portTICK_PERIOD_MS` 默认是 10ms（100Hz Tick）。如果业务需要 1ms 级精度，可以在 `sdkconfig` 中设置 `CONFIG_FREERTOS_HZ=1000`，然后 `Get_SysTick()` 可以直接返回 `xTaskGetTickCount()`。

#### 步骤 4：重写 `components/bsp/bsp_gpio.c`

```c
// bsp_gpio.c —— ESP32 版本
// 关键变化：用 ESP-IDF 的 gpio_config() API 替代寄存器直接操作

#include "bsp_gpio.h"
#include "driver/gpio.h"   // ESP-IDF GPIO 驱动
#include "esp_log.h"

// --- 不透明指针的新定义 ---
// STM32 版本存储的是 GPIO_TypeDef* 和 pin 位掩码
// ESP32 版本只需要存储 GPIO 编号（ESP32 的 GPIO 是独立编号的）
struct GPIO_Pin {
    gpio_num_t pin_num;    // ESP32 的 GPIO 编号：GPIO_NUM_0 ~ GPIO_NUM_39
    uint8_t    level;      // 当前电平（ESP32 没有 ODR 寄存器直接可读，需要缓存）
};

// --- 静态内存池（继续沿用这个模式） ---
static struct GPIO_Pin pool[16];
static uint8_t pool_idx = 0;

// --- 翻译函数：BSP 通用枚举 → ESP32 GPIO 编号 ---
// GPIO_NUM_0 到 GPIO_NUM_15 在 ESP-IDF 中的值恰好是 0~15
// 但 GPIOC13 在 STM32 上对应的是 GPIOC 的第 13 位
// 在 ESP32 上，我们需要知道每个端口对应的实际 GPIO 编号
// 这里假设我们使用的 ESP32 开发板的引脚映射：
//   BSP_PORT_A → GPIO 0~15
//   BSP_PORT_B → GPIO 16~31
//   BSP_PORT_C → GPIO 32~39（ESP32 只有 GPIO 0~39，其中 34~39 仅输入）
static void BSP_GPIO_Translate(BSP_Port_t bsp_port, BSP_Pin_t bsp_pin,
                               gpio_num_t* out_gpio_num) {
    // 默认的映射方案：平铺映射
    switch (bsp_port) {
        case BSP_PORT_A: *out_gpio_num = (gpio_num_t)(bsp_pin);        break;  // GPIO 0~15
        case BSP_PORT_B: *out_gpio_num = (gpio_num_t)(bsp_pin + 16);   break;  // GPIO 16~31
        case BSP_PORT_C: *out_gpio_num = (gpio_num_t)(bsp_pin + 32);   break;  // GPIO 32~39
        case BSP_PORT_D: *out_gpio_num = (gpio_num_t)(bsp_pin + 32);   break;  // 如果不够可以复用
        default: *out_gpio_num = GPIO_NUM_NC; break;  // Not Connected
    }

    // 如果你是具体的开发板（如 ESP32-DevKitC），可以在这里硬编码实际引脚映射：
    // if (bsp_port == BSP_PORT_C && bsp_pin == BSP_PIN_13) {
    //     *out_gpio_num = GPIO_NUM_2;  // 板载 LED 接在 GPIO2
    // }
}

// --- 构造函数 ---
struct GPIO_Pin* GPIO_Pin_Create(GPIO_Config_t* cfg) {
    if (!cfg) return NULL;
    if (pool_idx >= 16) return NULL;

    struct GPIO_Pin* obj = &pool[pool_idx++];
    gpio_num_t gpio_num;
    BSP_GPIO_Translate(cfg->port, cfg->pin, &gpio_num);

    obj->pin_num = gpio_num;

    // ESP-IDF 的 GPIO 配置（一次配置一个引脚）
    gpio_config_t io_conf = {0};

    // 设置引脚位掩码
    io_conf.pin_bit_mask = (1ULL << gpio_num);

    // 翻译模式
    switch (cfg->mode) {
        case BSP_GPIO_MODE_OUT_PP:
            io_conf.mode = GPIO_MODE_OUTPUT;
            // 禁用上拉下拉，使输出为推挽模式
            io_conf.pull_up_en = GPIO_PULLUP_DISABLE;
            io_conf.pull_down_en = GPIO_PULLDOWN_DISABLE;
            break;
        case BSP_GPIO_MODE_IN_PULLUP:
            io_conf.mode = GPIO_MODE_INPUT;
            io_conf.pull_up_en = GPIO_PULLUP_ENABLE;
            io_conf.pull_down_en = GPIO_PULLDOWN_DISABLE;
            break;
        case BSP_GPIO_MODE_IN_FLOATING:
            io_conf.mode = GPIO_MODE_INPUT;
            io_conf.pull_up_en = GPIO_PULLUP_DISABLE;
            io_conf.pull_down_en = GPIO_PULLDOWN_DISABLE;
            break;
        default:
            io_conf.mode = GPIO_MODE_OUTPUT;
            break;
    }

    io_conf.intr_type = GPIO_INTR_DISABLE;  // 不使用中断
    gpio_config(&io_conf);

    // 默认输出高电平
    if (cfg->mode == BSP_GPIO_MODE_OUT_PP) {
        gpio_set_level(gpio_num, 1);
        obj->level = 1;
    } else {
        obj->level = 1;  // 输入模式默认为高
    }

    return obj;
}

// --- 操作接口 ---

void GPIO_SetHigh(struct GPIO_Pin* pin) {
    if (pin) {
        gpio_set_level(pin->pin_num, 1);
        pin->level = 1;
    }
}

void GPIO_SetLow(struct GPIO_Pin* pin) {
    if (pin) {
        gpio_set_level(pin->pin_num, 0);
        pin->level = 0;
    }
}

void GPIO_Toggle(struct GPIO_Pin* pin) {
    if (pin) {
        pin->level = !pin->level;
        gpio_set_level(pin->pin_num, pin->level);
    }
}

uint8_t GPIO_GetLevel(struct GPIO_Pin* pin) {
    if (!pin) return 0;
    pin->level = gpio_get_level(pin->pin_num);
    return pin->level;
}
```

**STM32 vs ESP32 实现对比**：

| 要素 | STM32 实现 | ESP32 实现 |
|------|-----------|-----------|
| 不透明指针内容 | `GPIO_TypeDef* port` + `uint16_t pin`（位掩码） | `gpio_num_t pin_num`（引脚编号）+ `uint8_t level`（缓存） |
| 操作 GPIO | 直接写 ODR 寄存器 `pin->port->ODR ^= pin->pin` | 调用 API `gpio_set_level(pin->pin_num, level)` |
| 读 GPIO | 读 IDR 寄存器 `pin->port->IDR & pin->pin` | 调用 API `gpio_get_level(pin->pin_num)` |
| 初始化 | `RCC_APB2PeriphClockCmd()` + `GPIO_Init()` | `gpio_config()` 一次性设置 |
| 时钟控制 | 需要显式使能外设时钟 | ESP-IDF 自动管理 |

#### 步骤 5：重写 `components/bsp/bsp_uart.c`

```c
// bsp_uart.c —— ESP32 版本
#include "bsp_uart.h"
#include "driver/uart.h"
#include "freertos/FreeRTOS.h"
#include "freertos/queue.h"

// --- 不透明指针 ---
struct UART_Periph {
    uart_port_t          uart_num;      // ESP32 UART 编号：UART_NUM_0/1/2
    QueueHandle_t        rx_queue;      // FreeRTOS 队列，用于接收缓冲
    BSP_UART_RxCallback_t rx_callback;  // 回调函数
    TaskHandle_t         rx_task;       // 接收任务句柄
};

// --- 静态内存池 ---
static struct UART_Periph pool[3];
static uint8_t pool_idx = 0;
static struct UART_Periph* instances[3] = {0, 0, 0};

// --- 翻译函数 ---
static void BSP_UART_Translate(BSP_UART_t uart, uart_port_t* out_num) {
    switch (uart) {
        case BSP_UART_1: *out_num = UART_NUM_0; break;
        case BSP_UART_2: *out_num = UART_NUM_1; break;
        case BSP_UART_3: *out_num = UART_NUM_2; break;
        default: *out_num = UART_NUM_0; break;
    }
}

// --- 接收任务（在 FreeRTOS 任务中轮询 UART 事件） ---
static void uart_rx_task(void* arg) {
    struct UART_Periph* uart = (struct UART_Periph*)arg;
    uint8_t byte;
    while (1) {
        // 阻塞等待接收一个字节
        if (uart_read_bytes(uart->uart_num, &byte, 1, portMAX_DELAY) > 0) {
            if (uart->rx_callback) {
                uart->rx_callback(byte);
            }
        }
    }
}

// --- 构造函数 ---
struct UART_Periph* UART_Periph_Create(UART_Config_t* cfg) {
    if (!cfg || pool_idx >= 3) return NULL;

    struct UART_Periph* obj = &pool[pool_idx++];
    uart_port_t uart_num;
    BSP_UART_Translate(cfg->uart, &uart_num);
    obj->uart_num = uart_num;
    obj->rx_callback = NULL;

    // ESP-IDF UART 配置
    uart_config_t uc = {
        .baud_rate = (int)cfg->baudrate,
        .data_bits = (cfg->data_bits == BSP_UART_DATA_9) ? UART_DATA_9_BITS : UART_DATA_8_BITS,
        .parity    = (cfg->parity == BSP_UART_PARITY_ODD)  ? UART_PARITY_ODD  :
                     (cfg->parity == BSP_UART_PARITY_EVEN) ? UART_PARITY_EVEN : UART_PARITY_DISABLE,
        .stop_bits = (cfg->stop_bits == BSP_UART_STOP_2)   ? UART_STOP_BITS_2 :
                     (cfg->stop_bits == BSP_UART_STOP_1_5) ? UART_STOP_BITS_1_5 : UART_STOP_BITS_1,
        .flow_ctrl = UART_HW_FLOWCTRL_DISABLE,
        .source_clk = UART_SCLK_APB,
    };

    uart_param_config(uart_num, &uc);
    uart_set_pin(uart_num,
        (uart_num == UART_NUM_0) ? 1  : 17,  // TX pin（默认）
        (uart_num == UART_NUM_0) ? 3  : 16,  // RX pin（默认）
        UART_PIN_NO_CHANGE, UART_PIN_NO_CHANGE);
    uart_driver_install(uart_num, 256, 0, 0, NULL, 0);

    // 创建接收任务
    xTaskCreate(uart_rx_task, "uart_rx", 2048, obj, 5, &obj->rx_task);

    instances[cfg->uart] = obj;
    return obj;
}
```

```c
#include "freertos/task.h"

#include "app_lighting.h"

static void lighting_task(void* arg) {
    while (1) {
        App_Lighting_Task();
        vTaskDelay(pdMS_TO_TICKS(1));  // 1ms 轮询间隔
    }

    // 2. 初始化业务组件
    App_Lighting_Init();

    // 3. 创建 FreeRTOS 任务（替代裸机超级循环）
```

### 2.2.6 ESP32 移植风险评估

| 风险点 | 风险等级 | 缓解措施 |
|--------|---------|---------|
| FreeRTOS Tick 精度（默认 100Hz=10ms）不足以驱动 1ms 级超时检查 | 中 | 在 `sdkconfig` 中设置 `CONFIG_FREERTOS_HZ=1000` |
| ESP32 GPIO 34~39 仅输入，不能做输出 | 中 | 检查 `device_led.c` 中配置的引脚，确保不分配 GPIO34~39 给 LED |
| `Delay_us()` 从忙等变为 `ets_delay_us()` 后行为差异 | 低 | `ets_delay_us()` 也是忙等，行为兼容 |
| `Delay_ms()` 从忙等变为 `vTaskDelay()` 后行为差异 | 中低 | `vTaskDelay()` 是让出 CPU 的阻塞延时，对业务逻辑无影响，反而更好 |
| ESP32 启动时间较长（SPI Flash 初始化、Wi-Fi 协议栈等） | 低 | LED 初始化在 `app_main()` 中，晚于系统启动，无影响 |
| ESP-IDF 的 `gpio_config()` 不支持"推挽输出"这个概念 | 极低 | ESP32 的 GPIO 输出天然就是推挽模式 |
| 静态内存池在 FreeRTOS 环境下仍然安全 | 极低 | `static` 变量在 `.bss` 段，FreeRTOS 不管理此区域 |

### 2.2.7 ESP32 移植的最大收益

移植到 ESP32 后，除了"代码能跑"之外，还有额外收益：

1. **Delay_ms() 从忙等变为让出 CPU**：原来 `Delay_ms(500)` 会忙等 500ms，现在 `vTaskDelay()` 会让出 CPU 给其他任务——这在 ESP32 双核上尤其有价值。

2. **UART 接收从中断驱动变为任务驱动**：更稳定，不容易在高速通信时丢字节。ESP-IDF 的 UART 驱动自带 256 字节硬件 FIFO + 软件 Ring Buffer。

3. **自动获得 Wi-Fi/BLE 能力**：可以在 `app_main()` 中初始化 Wi-Fi，然后 `App_Lighting_Task()` 旁边再加一个 `App_MQTT_Task()`，实现手机远程控制 LED。App 层代码仍然可以保持硬件无关！

4. **OTA 固件升级**：ESP-IDF 自带 OTA 能力，项目可以直接从 1.0 升级到 2.0，不需要修改任何 App/Devices 层代码。

## 3.1 架构质量总评

| 评估维度 | 评分 | 核心论据 |
|---------|------|---------|
| **维护性** | ★★★★★ | 每类变更都有明确的入口文件；修改业务逻辑不需要理解硬件；修改硬件接线不需要翻遍代码 |
| **通用性** | ★★★★☆ | App/Devices 层完全硬件无关；BSP 接口通用但实现芯片相关；适合所有有 GPIO 概念的 MCU |
| **阅读性** | ★★★★★ | 语义化命名 + C99 指定初始化器 + 分层阅读路径 + 注释说"为什么" |
| **移植性** | ★★★★★ | 同架构芯片移植工作量 < 1 天；异构芯片移植上层零修改；BSP 是精准的隔离面 |

## 3.2 两种移植场景对比

| 对比维度 | STM32F103 → GD32F103 | STM32F103 → ESP32 |
|---------|---------------------|-------------------|
| **移植类型** | 同架构替代 | 跨架构移植 |
| **CPU 内核** | Cortex-M3 → Cortex-M3（同） | Cortex-M3 → Xtensa LX6（异） |
| **开发框架** | STM32 SPL → GD32 SPL（同类） | 裸机 → ESP-IDF + FreeRTOS（异类） |
| **App 层修改** | ✅ 零修改 | ✅ 零修改 |
| **Devices 层修改** | ✅ 零修改 | ✅ 零修改 |
| **BSP 头文件修改** | ✅ 零修改 | ✅ 零修改 |
| **BSP 实现修改** | 宏名替换（RCC→RCU 等） | 完全重写（ESP-IDF API） |
| **System 层修改** | 头文件 + SysTick 参数 | 重写（FreeRTOS Tick） |
| **中断处理修改** | 重命名文件 + 替换头文件 | 删除（FreeRTOS 任务替代） |
| **底层驱动修改** | 整体替换（STM32 SPL → GD32 SPL） | 全部删除（ESP-IDF 替代） |
| **main.c 修改** | NVIC 初始化函数名 | 重写为 FreeRTOS 任务模式 |
| **构建系统修改** | Keil 工程选项更新 | 全部重来（CMake + idf.py） |
| **需要修改的总文件数** | ~8 个（含目录替换） | ~6 个重写 + 4 个新增 CMakeLists |
| **上层不变的文件数** | 6 个（App + Devices + BSP headers） | 6 个（App + Devices + BSP headers） |
| **预计工作量** | 0.5~1 天 | 2~3 天 |
| **移植后额外收益** | 成本更低 | FreeRTOS + Wi-Fi + BLE + OTA |

## 3.3 这个架构教给我们的经验

```
                          ┌──────────────────────────────┐
                          │     分层架构的核心价值        │
                          │                              │
                          │  不是让代码更短、更快，      │
                          │  而是让变化的冲击范围最小化。 │
                          │                              │
                          │  变化是确定的（芯片会换、     │
                          │  需求会变、人员会流动），    │
                          │  我们能控制的是变化发生时     │
                          │  需要改多少代码。            │
                          └──────────────────────────────┘
```

| 经验 | 说明 |
|------|------|
| **画一条清晰的线** | 线的这边是"不变的东西"（业务逻辑、设备抽象），线的那边是"会变的东西"（芯片型号、驱动库）。BSP 层就是这条线。 |
| **接口是契约** | `bsp_gpio.h` 里的 4 个函数和 3 个枚举就是契约。所有上层代码只按契约办事，不关心契约是怎么履行的。 |
| **物理隔离胜过约定** | 不透明指针比"大家说好不直接访问寄存器"靠谱一万倍。编译器强制约束 > 团队口头约定。 |
| **一个文件只改一个理由** | 修改业务逻辑→改 App 层；修改硬件接线→改 Devices 层配置；换芯片→改 BSP 层实现。文件职责单一，修改理由就单一。 |
| **静态优于动态** | 静态内存池、编译期常量、枚举而非 `#define`——所有能在编译期确定的东西都不留到运行时。嵌入式系统的可靠性来源于确定性。 |
